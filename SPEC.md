# Pokenese — Technical & Functional Specification

## 1. Project Overview

Pokenese is a daily-game web application that teaches Mandarin Chinese through Pokémon names. Players are shown a Pokémon's Chinese name and must guess the English name. It combines the mechanics of Wordle/Connections with vocabulary acquisition, targeting adult learners who enjoy games.

## 3. Data Model

### 3.1 Pokémon Data (TypeScript Constant)

All Pokémon data lives in `frontend/src/data/pokemon.ts` as a typed constant array. This file is generated by a scraping script and can be manually edited.

```typescript
// frontend/src/types/pokemon.ts
export interface PokemonType {
  id: number;                         // National Pokédex number
  name_en: string;                    // English name e.g. "Charizard"
  name_zh: string;                    // Traditional Chinese characters e.g. "噴火龍"
  name_zh_simplified: string;         // Simplified Chinese e.g. "喷火龙"
  pinyin: string;                     // With tone marks e.g. "Pēn Huǒ Lóng"
  pinyin_numbered: string;            // Numbered tones e.g. "Pen1 Huo3 Long2"
  ipa: string;                        // IPA e.g. "/pʰən˥ xwɔ˨˩˦ lʊŋ˧˥/"
  etymology: EtymologyEntry[];        // Per-character breakdown
  generation: 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 | 9;
  type1: PokemonTypeName;
  type2: PokemonTypeName | null;
  category: string;                   // Pokédex category e.g. "Flame Pokémon"
  evolution_line: number[];           // National dex IDs of all mons in same evo line
  sprite_url: string;                 // URL to official sprite (PokeAPI CDN)
  audio_filename: string | null;      // e.g. "006.mp3" if pre-recorded exists
}

export interface EtymologyEntry {
  character: string;                  // Single character e.g. "噴"
  pinyin: string;                     // e.g. "pēn"
  meaning: string;                    // English meaning e.g. "to spit/spray"
}

export type PokemonTypeName =
  | 'Normal' | 'Fire' | 'Water' | 'Electric' | 'Grass' | 'Ice'
  | 'Fighting' | 'Poison' | 'Ground' | 'Flying' | 'Psychic' | 'Bug'
  | 'Rock' | 'Ghost' | 'Dragon' | 'Dark' | 'Steel' | 'Fairy';
```

**Data sources for scraping script (`backend/scripts/scrape_pokemon.py`):**
- English names + types + generation + category: PokéAPI (`pokeapi.co`)
- Chinese names + pinyin + etymology: Bulbapedia Chinese name pages, supplemented by `zh.bulbapedia.wikia.com`
- IPA: Derived from pinyin using a pinyin-to-IPA conversion library (`pinyin` Python package)
- Evolution lines: PokéAPI evolution chain endpoint
- Sprites: PokeAPI sprite CDN URLs

**Script output:** Writes `frontend/src/data/pokemon.ts` as a valid TypeScript file with `export const POKEMON_DATA: PokemonType[] = [...]`.

### 3.2 Database Schema (PostgreSQL)

```sql
-- Users
CREATE TABLE users (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email       TEXT UNIQUE NOT NULL,
  username    TEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Daily challenges (pre-seeded by date)
CREATE TABLE daily_challenges (
  date        DATE PRIMARY KEY,
  pokemon_id_1 INT NOT NULL,  -- National dex IDs
  pokemon_id_2 INT NOT NULL,
  pokemon_id_3 INT NOT NULL
);

-- Daily results per user
CREATE TABLE daily_results (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID REFERENCES users(id) ON DELETE CASCADE,
  date         DATE NOT NULL,
  challenge_num SMALLINT NOT NULL CHECK (challenge_num IN (1, 2, 3)),
  guesses      TEXT[] NOT NULL,       -- Ordered list of guessed english names
  score        INT NOT NULL,
  hints_used   SMALLINT NOT NULL,
  completed_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (user_id, date, challenge_num)
);

-- Challenge mode state
CREATE TABLE challenge_state (
  user_id      UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  seen_ids     INT[] NOT NULL DEFAULT '{}',  -- Pokemon IDs already seen
  total_score  INT NOT NULL DEFAULT 0,
  is_active    BOOLEAN NOT NULL DEFAULT true,
  updated_at   TIMESTAMPTZ DEFAULT NOW()
);

-- Challenge mode results (individual pokemon results within a run)
CREATE TABLE challenge_results (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID REFERENCES users(id) ON DELETE CASCADE,
  pokemon_id   INT NOT NULL,
  guesses      TEXT[] NOT NULL,
  score        INT NOT NULL,
  hints_used   SMALLINT NOT NULL,
  run_number   INT NOT NULL,  -- Increments each time state is reset
  completed_at TIMESTAMPTZ DEFAULT NOW()
);

-- Glossary (unlocked pokemon per user)
CREATE TABLE glossary_entries (
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
  pokemon_id  INT NOT NULL,
  unlocked_at TIMESTAMPTZ DEFAULT NOW(),
  times_seen  INT NOT NULL DEFAULT 1,
  PRIMARY KEY (user_id, pokemon_id)
);

-- Refresh tokens
CREATE TABLE refresh_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
  token_hash  TEXT NOT NULL,
  expires_at  TIMESTAMPTZ NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

## 4. Anonymous User State (localStorage)

For users without accounts, all state is stored in `localStorage` under the key `pokenese_state`. Schema:

```typescript
interface LocalState {
  version: number;                    // Schema version for migrations
  glossary: number[];                 // Unlocked pokemon IDs
  glossary_seen_count: Record<number, number>; // pokemon_id -> times seen
  daily: {
    [date: string]: {                 // ISO date key e.g. "2026-05-25"
      challenge_1?: ChallengeResult;
      challenge_2?: ChallengeResult;
      challenge_3?: ChallengeResult;
    }
  };
  challenge: {
    seen_ids: number[];
    total_score: number;
    is_active: boolean;
    run_number: number;
  };
  settings: UserSettings;
}

interface ChallengeResult {
  guesses: string[];
  score: number;
  hints_used: number;
  completed_at: string;              // ISO timestamp
}

interface UserSettings {
  show_pinyin: boolean;              // Default: true
  show_ipa: boolean;                 // Default: false
  extreme_mode: boolean;             // Hides pinyin AND chinese, shows only audio. Default: false
  auto_play_audio: boolean;          // Default: false
  theme: 'dark' | 'light';          // Default: 'dark'
}
```

When a user registers, the frontend POSTs the full `LocalState` to `POST /api/auth/register` and the backend migrates this data to the database.

## 5. Game Logic

### 5.1 Challenge Structure

A single challenge proceeds as follows:

1. **Initial state:** Display the Pokémon's Chinese name (hanzi + pinyin, toggleable). Audio button available. Zero hints shown.
2. **Player submits a guess** (English Pokémon name from autocomplete).
3. **If correct:** Show success screen, calculate score, unlock Pokémon in glossary.
4. **If wrong (guess attempt 1–4):** Reveal the next hint in sequence, show the wrong guess in history.
5. **If all 5 guesses exhausted:** Reveal Pokémon, calculate proximity score, unlock Pokémon in glossary.

**Hint sequence (revealed after each wrong guess):**

| Wrong Guess # | Hint Revealed |
|---|---|
| 1 | Character meanings (etymology breakdown) |
| 2 | Generation |
| 3 | Type(s) |
| 4 | Pokédex category (e.g. "The Flame Pokémon") |

### 5.2 Scoring

**On success (guessed correctly within 5 attempts):**
```
score = 3000 - (hints_shown * 600)
```

| Guesses used | Hints shown | Score |
|---|---|---|
| 1 (no hints) | 0 | 3000 |
| 2 | 1 | 2400 |
| 3 | 2 | 1800 |
| 4 | 3 | 1200 |
| 5 | 4 | 600 |

**On failure (5 wrong guesses):**

Evaluated against player's *best guess* (closest to correct):

| Condition | Score |
|---|---|
| Best guess is in the same evolution line | 300 |
| Best guess shares all types with correct | 200 |
| Best guess shares one type with correct | 100 |
| Best guess shares a Chinese character with correct | 50 |
| None of the above | 0 |

Only the highest applicable score is awarded (not cumulative).

**Proximity evaluation order:** evolution line > full type match > partial type match > shared Chinese character.

### 5.3 Daily Mode

- Each UTC calendar day maps to a set of exactly 3 Pokémon.
- Pre-seeded in the `daily_challenges` table for at least 1 year in advance.
- All 3 challenges are presented sequentially in the same session.
- Players cannot skip ahead; challenge 2 is locked until challenge 1 is completed.
- Total daily score = sum of scores from all 3 challenges (max 9000).
- After all 3 are complete, a share graphic is generated.
- **No more attempts after completion** — the result is final for that day.
- A countdown timer shows time until the next daily challenge.

### 5.4 Challenge Mode

- Pokemon are drawn from the full Pokédex (all generations) in pseudo-random order.
- Each Pokémon is presented at most once per run (until the full Pokédex is covered, then it resets — congratulations screen shown).
- If a player scores 0 on any challenge (complete failure, no proximity points either — i.e. 5 wrong guesses with no proximity), their entire run resets: `seen_ids = []`, `total_score = 0`, `run_number += 1`.
  - **Clarification on reset trigger:** Reset occurs when a player exhausts all 5 guesses and receives a score of 0 (no proximity match). A proximity score (even 50 points) does NOT trigger a reset.
- Score accumulates across all challenges in a run.
- Progress is shown as "X / 1025" (or total Pokémon count).

## 11. Audio Implementation

1. **Primary:** Use the browser's Web Speech API (`SpeechSynthesis`) with a Mandarin Chinese voice (`lang: 'zh-CN'`).
   - Speak the `name_zh_simplified` field (simplified Chinese is better supported by TTS engines).
   - If Mandarin voice unavailable, fall back to pinyin text display.
2. **Enhancement (optional):** Pre-recorded `.mp3` files in `frontend/public/audio/` named by dex number (e.g. `006.mp3`). If `audio_filename` is set on the Pokémon, use the file; otherwise fall back to TTS.
3. The `useAudio` hook handles this logic transparently.

---

## 12. Glossary Unlock Rules

- A Pokémon is added to the glossary the **first time a player encounters it** in either Daily or Challenge mode, regardless of whether they guess correctly.
- Subsequent encounters increment `times_seen`.
- The glossary persists across sessions (localStorage or database).

---

## 13. Daily Challenge Seeding

The `seed_daily.py` script:
1. Loads all Pokémon IDs from the TypeScript data file.
2. For each future date (e.g. 1 year out), pseudo-randomly selects 3 Pokémon IDs using a seeded RNG (seed = date string + secret salt).
3. Ensures no Pokémon repeats within a 7-day window.
4. Inserts rows into `daily_challenges`.

The seed algorithm is **deterministic** given the same date and salt, so the script can be re-run safely (upsert on conflict).

---

## 14. Guess Autocomplete

- The autocomplete list is sourced entirely from the client-side TypeScript constants (no API call needed).
- Input matching: case-insensitive prefix + substring match.
- Dropdown shows Pokémon sprite, English name, and Chinese name.
- Maximum 8 suggestions shown at a time.
- Previously guessed Pokémon in the current challenge are excluded from suggestions.

---

## 15. Share Graphic Generation

- Client-side using `html2canvas` to render a hidden `<div>` to a `<canvas>` then export as PNG.
- The share `<div>` (hidden off-screen) contains the score grid, date, username, and branding.
- User can download or share via Web Share API (mobile) or copy to clipboard.
- The graphic format is documented in `VISUAL_DESIGN_SPEC.md`.

---

## 16. Streak Calculation

- Daily streak: consecutive days where the user completed all 3 daily challenges before midnight UTC.
- Streak is stored and recomputed on each daily completion.
- A missed day resets streak to 0.

---

## 17. Performance Requirements

- Initial page load: < 3s on 4G mobile.
- Pokémon data JS bundle: The `pokemon.ts` file will be large (~1MB). It should be imported dynamically (`next/dynamic` or `import()`) only when needed (game screen, autocomplete).
- API responses: < 200ms p95 for all endpoints.
- Autocomplete filter: < 16ms (synchronous, in-memory).

---

## 18. Pokémon Coverage

Target: All Pokémon as of Generation IX (National Dex #1–1025, including regional forms where they have distinct Chinese names). Mega evolutions and Gigantamax forms are excluded from gameplay but may be noted in glossary.

## 19. Non-Goals (Explicit Out of Scope)

- Multiplayer / real-time features
- In-app purchases or monetization
- Native mobile apps (iOS/Android)
- Other languages beyond English (UI) and Mandarin (Pokémon names)
- Admin dashboard (daily challenges managed directly via DB scripts)
